### 同类搜索工具代码重复问题

所有搜索类 Tool（SearchBomTool、SearchEquipmentTool、SearchLineTool、SearchProcessTool、SearchTeamTool、SearchWorkStepTool）均使用相同模式：

1. 调用 Service 获取全部数据
2. 内存 filter 按关键词过滤
3. 返回 `Map.of("count", ..., "items", ...)`

**建议**：提取一个通用的 `AbstractSearchTool` 基类或函数式模板，减少重复代码，并统一处理 null 安全、空数据、分页限制等问题。

### 改进建议（不修改源代码）

1. **安全增强**：建议 AiChatController 的 `extractUserId` 方法在无法提取 token 时返回 `401` 错误而非创建 userId=0 的会话
2. **性能优化**：所有搜索类工具应从数据库层模糊搜索（SQL `LIKE`），而非全量加载后内存过滤，避免大数据量场景下的性能问题
3. **Provider 适配**：AiAgentServiceImpl 中的 System Prompt 应考虑按 provider 动态适配
4. **单元测试覆盖率**：已有 7 类共 78 个测试方法，覆盖了核心链路。建议补充 AiAgentServiceImpl 的单元测试（当前为全流程集成逻辑，Mock 依赖多但值高）、AiChatController 的单元测试、ClaudeProvider/OpenAiProvider 的 API 响应解析单元测试
5. **限流计数器清理**：ToolExecutor 中的 `sessionRateLimiters` ConcurrentHashMap 会持续增长。建议增加定时清理过期 session 计数器的任务
6. **API Key 安全性**：application-ai.yml 中 api-key 通过环境变量注入是安全的做法，但建议在 ClaudeProvider/OpenAiProvider 中添加日志脱敏（当前的异常消息可能泄露 API Key 的部分内容）
7. **确认语义**：`detectConfirmation()` 只检测单条最近 user 消息中的关键词，不够严谨。若用户在包含确认关键词后继续追加了其他消息，确认判断可能不准确。建议改为状态机模型——当需要确认时设置 `pendingConfirmation` 状态，下一轮对话仅检测确认语义
